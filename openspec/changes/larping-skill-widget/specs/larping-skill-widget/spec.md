# Larping Skill Widget

**Owned by**: LarpingApp (app-specific dashboard widget)

## Purpose
Provide a LarpingApp-specific dashboard widget that visualizes skill usage distribution across characters using data from OpenRegister's GraphQL faceting API. This widget is part of LarpingApp's dashboard experience, following the cross-app dashboard patterns defined in the `built-in-dashboards` spec and using the `CnDashboardPage` shared component from `@conduction/nextcloud-vue`.

## Requirements

### Requirement: The dashboard MUST display a skill usage pie chart
The LarpingApp dashboard MUST include a donut chart showing the distribution of skills across all characters, powered by data from OpenRegister's GraphQL faceting API. The chart uses the `facets` capability of the GraphQL `character` query to aggregate skill counts server-side, avoiding the need for client-side data processing of raw UUID arrays.

#### Scenario: Skill usage chart with data
- **GIVEN** 5 characters exist in OpenRegister with skills assigned:
  - "Aldric" with skills: ["Swordsmanship", "Shield Bash"]
  - "Elara" with skills: ["Healing", "Swordsmanship"]
  - "Grimm" with skills: ["Swordsmanship", "Alchemy", "Healing"]
  - "Thorne" with skills: ["Archery", "Stealth"]
  - "Lyra" with skills: ["Healing", "Alchemy"]
- **WHEN** the user views the LarpingApp dashboard
- **THEN** a donut chart MUST display with skills ranked by popularity:
  - "Swordsmanship" = 3 characters
  - "Healing" = 3 characters
  - "Alchemy" = 2 characters
  - "Shield Bash" = 1 character
  - "Archery" = 1 character
  - "Stealth" = 1 character
- **AND** the chart MUST have the widget title "Skill Usage by Characters" as defined in `widgetDefs`

#### Scenario: Skill usage chart with many skills shows top 10
- **GIVEN** characters reference more than 10 distinct skills
- **WHEN** the chart renders
- **THEN** only the top 10 skills by popularity MUST be shown as individual slices
- **AND** remaining skills MUST be grouped into an "Other" slice with their combined count
- **AND** the slices MUST be sorted descending by count before the top-10 cutoff

#### Scenario: Skill usage chart with no data
- **GIVEN** no characters exist or no characters have skills assigned
- **WHEN** the user views the dashboard
- **THEN** the chart area MUST display a message "No skill data available"
- **AND** no empty chart MUST be rendered (the `VueApexCharts` component MUST NOT mount)

#### Scenario: Chart respects Nextcloud theme
- **GIVEN** the user has Nextcloud dark mode enabled (detected via `document.body.dataset.themeDark`)
- **WHEN** the chart renders
- **THEN** chart colors MUST be visible against the dark background
- **AND** labels MUST use appropriate contrast colors
- **AND** the ApexCharts `theme.mode` MUST be set to `'dark'`
- **AND** chart background MUST be `'transparent'` to inherit the dashboard card background

#### Scenario: Chart respects system color scheme when no explicit theme
- **GIVEN** the user has not set an explicit Nextcloud theme
- **AND** the OS-level `prefers-color-scheme` is `dark`
- **AND** `document.body.dataset.themeLight` is NOT set
- **WHEN** the chart renders
- **THEN** the chart MUST use dark mode styling via `window.matchMedia('(prefers-color-scheme: dark)')`

### Requirement: The widget MUST fetch data via a single GraphQL faceting query
The skill usage data MUST be retrieved using a single GraphQL query that leverages OpenRegister's faceting capability on the `skills` field. This replaces the earlier two-query approach (characters + skill name resolution) with a server-side aggregation that returns pre-resolved labels.

#### Scenario: Single faceted query fetches skill distribution
- **GIVEN** the LarpingApp has characters stored in OpenRegister with a `skills` array property
- **WHEN** the dashboard loads and the SkillUsageChart component mounts
- **THEN** the widget MUST execute the GraphQL query:
  ```graphql
  {
    character(first: 1, facets: ["skills"]) {
      totalCount
      facets
    }
  }
  ```
- **AND** the `facets` response MUST contain a `skills.data.buckets` array
- **AND** each bucket MUST have a `label` (or `value`) and a `count`

#### Scenario: GraphQL requests include Nextcloud authentication
- **GIVEN** the user is logged into Nextcloud
- **WHEN** the GraphQL query executes via the `queryGraphQL()` utility from `services/graphql.js`
- **THEN** the request MUST POST to the URL generated by `generateUrl('/apps/openregister/api/graphql')`
- **AND** the request MUST include the `requesttoken` header from `getRequestToken()` (via `@nextcloud/auth`)
- **AND** credentials MUST be set to `'same-origin'`

#### Scenario: GraphQL query handles rate limiting
- **GIVEN** the OpenRegister GraphQL endpoint returns HTTP 429
- **WHEN** the widget processes the response
- **THEN** the widget MUST throw an error including the `Retry-After` header value
- **AND** the error MUST display "Rate limited -- retry after {N} seconds"

#### Scenario: GraphQL query handles authentication failure
- **GIVEN** the user's session has expired
- **WHEN** the GraphQL query returns HTTP 401
- **THEN** the widget MUST display "Authentication error -- please log in again"

#### Scenario: GraphQL query handles errors gracefully
- **GIVEN** the OpenRegister GraphQL endpoint returns a GraphQL error response or is unavailable
- **WHEN** the widget tries to fetch data
- **THEN** the widget MUST display the error message from the response in the `.chart-error` container
- **AND** a "Retry" button (NcButton type="secondary") MUST be available that re-invokes `fetchData()`
- **AND** the error text MUST be styled with `var(--color-error)` color

### Requirement: The widget MUST aggregate skill counts from facet buckets
Skill popularity counts MUST be extracted from the GraphQL faceting response, which provides pre-aggregated server-side counts.

#### Scenario: Process facet buckets into chart data
- **GIVEN** the GraphQL response contains `data.character.facets.skills.data.buckets` as an array of `{label, value, count}` objects
- **WHEN** the widget processes the response
- **THEN** it MUST sort buckets descending by `count`
- **AND** it MUST use `label` (falling back to `value`) for the chart label of each bucket
- **AND** it MUST use `count` for the chart series value of each bucket

#### Scenario: Handle empty facet response
- **GIVEN** the GraphQL response has an empty `buckets` array or `facets.skills` is missing
- **WHEN** the widget processes the data
- **THEN** `skillLabels` and `skillCounts` MUST both be set to empty arrays
- **AND** the `hasData` computed property MUST return `false`
- **AND** the "No skill data available" message MUST display

#### Scenario: Handle missing character data in response
- **GIVEN** the GraphQL response returns `data.character` as null or undefined
- **WHEN** the widget processes the data
- **THEN** the widget MUST gracefully set empty arrays for labels and counts
- **AND** MUST NOT throw a JavaScript error

- **AND** the widget definition MUST have `id: 'skill-usage'` in the `widgetDefs` array

#### Scenario: Widget layout is user-customizable

- **WHEN** the user repositions or resizes the Skill Usage widget

#### Scenario: Widget has consistent card styling with other dashboard widgets

- **WHEN** the chart renders
- **THEN** it MUST respect the card background provided by the CnDashboardPage component

### Requirement: The widget MUST display a character stat breakdown panel
The dashboard MUST include a widget that shows the calculated ability scores for a selected character, displaying the base value, applied modifiers from skills/items/conditions/events, and the final computed value. This mirrors the server-side calculation performed by `CharacterService.calculateCharacter()`.

#### Scenario: Display ability scores for a character
- **GIVEN** a character "Aldric" exists with skills ["Swordsmanship"] and items ["Iron Shield"]
- **AND** the ability "Strength" has a base value of 10
- **AND** skill "Swordsmanship" has an effect that applies modifier +3 (positive) to "Strength"
- **AND** item "Iron Shield" has an effect that applies modifier +1 (positive) to "Strength"
- **WHEN** the user selects "Aldric" in the character stat widget
- **THEN** the widget MUST display:
  - Ability: "Strength"
  - Base: 10
  - Final: 14
  - Breakdown: "+3 from Swordsmanship (effect), +1 from Iron Shield (effect)"

#### Scenario: Display negative modifiers distinctly
- **GIVEN** a character has a condition "Cursed" with an effect that applies modifier 2 (negative) to "Agility"
- **AND** "Agility" has base value 8
- **WHEN** the stat breakdown renders
- **THEN** the widget MUST show the modifier as "-2 from Cursed (condition)"
- **AND** the final value MUST be 6
- **AND** negative modifiers MUST be styled with `var(--color-error)` text color

#### Scenario: Character with no skills or items
- **GIVEN** a character exists with empty `skills`, `items`, `conditions`, and `events` arrays
- **WHEN** the stat breakdown renders
- **THEN** all abilities MUST show their base value as the final value
- **AND** the breakdown column MUST show "No modifiers"

### Requirement: The widget MUST display an effect audit trail for each ability
Each ability score in the character stat widget MUST have an expandable audit trail that shows every effect that was applied, in the order they were applied, matching the `audit` array produced by `CharacterService.applyModifierToAbility()`.

#### Scenario: Expand audit trail for an ability
- **GIVEN** a character's "Strength" ability has the following audit entries:
  - `{ type: "effect", effect: { name: "Blade Training", modifier: 3, modification: "positive" }, old: 10, new: 13 }`

- **WHEN** the user clicks on the "Strength" ability row
- **THEN** an expandable section MUST display the audit entries as a timeline:
  - Entry 1: "Blade Training: 10 -> 13 (+3)"
  - Entry 2: "Heavy Armor Penalty: 13 -> 12 (-1)"
- **AND** each entry MUST show the old value, new value, and modifier direction

#### Scenario: Audit trail for ability with no effects
- **GIVEN** an ability has an empty `audit` array
- **WHEN** the user clicks on the ability row
- **THEN** the expandable section MUST show "No effects applied to this ability"

## Nextcloud Integration Analysis

**Status**: Partially implemented. The `SkillUsageChart.vue` component and `DashboardIndex.vue` with `CnDashboardPage` integration exist in the codebase. The skill usage chart uses GraphQL faceting for server-side aggregation. Character stat display, effect chain visualization, multi-character comparison, skill dependency graph, interactive skill selection, and character sheet widgets are not yet implemented.

**Nextcloud Core Interfaces**:
- `IDashboardWidget` / `IAPIWidgetV2` (`OCP\Dashboard`): Implement a `SkillUsageWidget` class in LarpingApp that registers with Nextcloud's dashboard framework. Use `IAPIWidgetV2` for async data loading -- the widget fetches skill usage data via a single GraphQL faceting query and renders a donut chart. This makes the widget available on Nextcloud's main dashboard page alongside other app widgets.
- `IBootstrap` (`OCP\AppFramework\Bootstrap\IBootstrap`): Register the widget during LarpingApp's bootstrap phase via `$context->registerDashboardWidget(SkillUsageWidget::class)`. This ensures the widget appears in Nextcloud's dashboard widget picker.
- `IInitialState` (`OCP\IInitialState`): Pass LarpingApp's OpenRegister configuration (register, character_schema, skill_schema, ability_schema) to the frontend via initial state, so the Vue widget component knows which GraphQL queries to construct without additional API calls.
- `IAppConfig`: Read LarpingApp's data source configuration (`character_source`, `skill_source`) to determine whether to show the widget or display a configuration message. The `RegisterObjectFetcher` already uses this to resolve register/schema pairs per entity type.

**Implementation Approach**:
- The existing `SkillUsageChart.vue` executes a single GraphQL faceting query (`character(first: 1, facets: ["skills"])`), extracts `facets.skills.data.buckets`, sorts by count, and renders via VueApexCharts donut chart. This is the primary implemented widget.
- The `DashboardIndex.vue` uses `CnDashboardPage` from `@conduction/nextcloud-vue` with a grid-based layout system (`DEFAULT_LAYOUT`). Widgets are defined in `widgetDefs` and rendered via scoped slots (`#widget-{id}`). The layout is customizable via drag-and-drop with the `layout-change` event.
- The `CharacterService.calculateCharacter()` method provides the stat calculation engine: it initializes ability scores from base values (`initializeAbilityScores`), then applies effects from skills, items, conditions, and events in sequence via `applyEntityEffects()`. Each effect applies a `modifier` value as either `positive` (add) or `negative` (subtract) to target abilities, recording an audit trail entry with `old`, `new`, `type`, and `effect` data.
- The `RegisterObjectFetcher` resolves entity types to OpenRegister register/schema pairs via `IAppConfig`, supporting the 9 entity types: character, player, ability, skill, item, condition, effect, event, setting.
- All entity data flows through the `useObjectStore()` Pinia store, which registers object types with `registerObjectType(slug, schemaId, registerId)` during `initializeStores()`.
- The GraphQL utility (`services/graphql.js`) handles authentication via `@nextcloud/auth` request tokens and includes error handling for 401, 429, and GraphQL-level errors.
- Widget styling uses Nextcloud CSS custom properties exclusively (`var(--color-primary-element)`, `var(--color-text-maxcontrast)`, `var(--color-background-hover)`, `var(--color-error)`, `var(--border-radius)`) for NL Design System theming compatibility.
- The `CharactersController.downloadPdf()` endpoint delegates PDF generation to DocuDesk's `PdfService` and `TemplateService`, producing character sheet PDFs named `{name}_character_sheet.pdf`.

**Dependencies on Existing OpenRegister Features**:
- GraphQL API (`/api/graphql`) with faceting support -- data source for character skill distribution queries.
- OpenRegister `ObjectService.getMapper()` -- used by `RegisterObjectFetcher` to resolve entity type to register/schema mapper.
- LarpingApp's OpenRegister configuration -- 9 entity types each with `{type}_register` and `{type}_schema` app config values, plus a shared `register` value.
- LarpingApp's dashboard grid infrastructure via `CnDashboardPage` -- layout container supporting drag-and-drop widget positioning.
- DocuDesk integration (optional) -- PDF export for character sheets via `PdfService.renderPdf()`.

**Dependencies on LarpingApp Domain Logic**:
- `CharacterService` -- server-side stat calculation engine with effect application order (skills -> items -> conditions -> events) and audit trail generation.
- `RegisterObjectFetcher` -- cross-app service bridge between LarpingApp and OpenRegister's `ObjectService`.
- Entity model relationships: Characters have arrays of skill/item/condition/event UUIDs. Skills/items/conditions have arrays of effect UUIDs. Effects have `abilities` arrays, `stat_id`, `modifier` (int), and `modification` ("positive"/"negative").
